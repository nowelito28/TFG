\section{Prototipos e Implementaciones}

El desarrollo principal del proyecto se articuló mediante varios prototipos de \textit{LKM} construidos progresivamente y apoyados en las funcionalidades validadas en los anteriores.
Esta estrategia permitió avanzar desde una primera toma de contacto con la programación en \textit{Espacio del Kernel} hasta la construcción del prototipo final capaz de generar un listado autenticado de los procesos activos del sistema desde dicha región privilegiada.

\subsection{\textit{Hello World}}
\label{sec:proto-helloworld}

El primer paso del desarrollo consistió en construir un \textit{LKM} básico cuyo objetivo era verificar cómo funciona el flujo básico de trabajo asociado a los \textit{Módulos del Kernel}.\\

Para ello, se diseñó inicialmente el fichero de compilación externa \textit{Makefile} basado en \textit{kbuild}, que delega el proceso de compilación en el sistema de construcción del propio \textit{Kernel} y se reutiliza en la compilación de los prototipos posteriores.
En este fichero se define la variable \verb|obj-m|, que indica a \textit{kbuild} qué fichero objeto (\texttt{.o}) debe compilarse como módulo cargable (\texttt{.ko}).
Por otro lado, se declaran dos reglas principales que estructuran el flujo de compilación. La primera, \verb|all|, invoca el sistema de construcción del \textit{Kernel} activo mediante la ruta \verb|/lib/modules/$(shell uname -r)/build|, y especifica la ruta del código fuente del módulo a compilar en el directorio actual con \verb|M=$(shell pwd)|, por lo que para realizar la compilación es necesario situar el \textit{Makefile} en el mismo directorio que el código fuente del módulo y ejecutar la compilación desde ese mismo directorio.
La regla \verb|clean|, como segunda regla, elimina los ficheros intermedios generados durante la compilación.
Por último, se declaran dichas reglas como \textit{phony targets} para evitar conflictos con ficheros del mismo nombre.
De este modo, el módulo puede compilarse de manera externa sin necesidad de modificar ni recompilar el núcleo del sistema.

\begin{code}[h]
\begin{lstlisting}[language=make]
obj-m := module_name.o

all: aux.sh
	make -C /lib/modules/$(shell uname -r)/build M=$(shell pwd) modules

clean:
	make -C /lib/modules/$(shell uname -r)/build M=$(shell pwd) clean

.PHONY: all clean
\end{lstlisting}
\caption[\textit{Makefile} genérico empleado para compilar \textit{LKM}s.]{\textit{Makefile} genérico empleado para compilar \textit{LKM}s.}
\label{cod:makefile}
\end{code}

En caso de ser necesario proporcionar alguna dependencia previa a la compilación, como un fichero auxiliar, esta puede declararse en la regla \verb|all|, como se muestra en el código anterior con el fichero \texttt{aux.sh}.
En este prototipo inicial no fue necesario hacer uso de esta característica, por lo que la dependencia mostrada debe entenderse únicamente como un ejemplo general de cómo declararla en caso de que fuera necesario.\\

Una vez definida la herramienta de compilación mediante \textit{Makefile} y \textit{kbuild}, se implementó un módulo simple, \verb|hello_world.c|, con el fin de validar el ciclo de vida básico de un \textit{LKM}: carga del módulo en el \textit{Kernel}, ejecución de funciones controladoras de carga y descarga del módulo, registro de mensajes en el \textit{log} del \textit{Kernel} y posterior descarga del módulo.\\

En primer lugar, se incluyeron las cabeceras necesarias para compilar el módulo y se declararon los metadatos básicos del componente:

\newpage

\begin{code}[h]
\begin{lstlisting}[language=C]
#include <linux/module.h>
#include <linux/kernel.h>
#include <linux/init.h>

MODULE_LICENSE("Dual BSD/GPL");
MODULE_AUTHOR("Noel Rodriguez Perez");
MODULE_DESCRIPTION("A simple Hello world LKM!");
MODULE_VERSION("0.1");
\end{lstlisting}
\caption[Cabeceras y metadatos del módulo \textit{Hello World}]{Cabeceras y metadatos del módulo \textit{Hello World}.}
\label{cod:hello_headers}
\end{code}

En este fragmento se incorporan las cabeceras necesarias para programar un \textit{LKM} de \textit{Linux} y, por otro lado, se definen varios metadatos descriptivos del componente, visibles posteriormente mediante la herramienta \texttt{modinfo}.
Entre ellos, la licencia resulta especialmente importante, debido a que condiciona el comportamiento del módulo dentro del \textit{Kernel} y determina si este queda marcado como \textit{tainted} al cargarse, no considerándose completamente limpio a efectos de depuración y soporte.
En este caso, se elige una licencia dual \textit{BSD/GPL}.
Por un lado, la compatibilidad con \textit{GPL}, empleada por el propio \textit{Kernel} de \textit{Linux}, evita que el módulo quede marcado como \textit{tainted}.
Por otro lado, la licencia \textit{BSD}, de carácter permisivo, permite el uso del módulo en entornos privativos sin tener que compartir el código fuente.\\

A continuación, el módulo necesita implementar sus funciones de inicialización y salida:

\begin{code}[h]
\begin{lstlisting}[language=C]
static int __init hello_start(void)
{
    printk(KERN_INFO "Loading hello world module...\n");
    printk(KERN_INFO "Hello world!!\n");
    return 0;
}

static void __exit hello_end(void)
{
    printk(KERN_INFO "Goodbye Mr.\n");
}
\end{lstlisting}
\caption[Funciones de inicialización/carga y salida/descarga del módulo \textit{Hello World}]{Funciones de inicialización/carga y salida/descarga del módulo \textit{Hello World}.}
\label{cod:hello_init_exit}
\end{code}

La función \verb|hello_start| se ejecuta al cargar el módulo en el \textit{Kernel} con \verb|insmod <module_name>.ko|. En este prototipo, su única finalidad es escribir mensajes en el \textit{buffer} de \textit{logs} del \textit{Kernel} mediante la función \verb|printk|, registrados también en el fichero \texttt{/var/log/kern.log}, utilizada como mecanismo habitual de salida de depuración y trazabilidad en \textit{Espacio del Kernel}, pudiendo elegir el nivel de prioridad de los mensajes.
Análogamente, la función \verb|hello_end| se ejecuta al descargar el módulo con \verb|rmmod <module_name>| y permite comprobar que el ciclo de vida del componente se completa correctamente, verificando los \textit{logs} del \textit{Kernel} con \verb|dmesg|.\\

Finalmente, ambos controladores deben asociarse explícitamente al flujo de control del módulo:

\begin{code}[h]
\begin{lstlisting}[language=C]
module_init(hello_start);
module_exit(hello_end);
\end{lstlisting}
\caption[Asociación de los puntos de entrada/carga y salida/descarga del módulo]{Asociación de los puntos de entrada/carga y salida/descarga del módulo.}
\label{cod:hello_module_init}
\end{code}

Estas macros indican al \textit{Kernel} qué función debe ejecutarse durante la carga del módulo y cuál debe invocarse durante su descarga.
Con ello queda definida la estructura mínima funcional de un \textit{LKM} y puede procederse a validar su correcto funcionamiento.\\

Con permisos de administrador (\textit{root}), se compiló el módulo, se cargó en el \textit{Kernel}, se inspeccionaron las trazas registradas en el log del \textit{Kernel}, se consultaron los metadatos del propio módulo y, finalmente, se descargó verificando también la ejecución de la rutina de salida:

\subsection{Fichero virtual \texttt{/proc/fdev}}
\label{sec:proto-proc-fdev}

El siguiente paso consistió en dotar al módulo de una interfaz de comunicación entre \textit{Espacio de Usuario} y \textit{Espacio del Kernel}.
Para ello, se recurrió al sistema de ficheros virtual \texttt{/proc} para exponer información del \textit{Kernel} y permitir la interacción controlada con el \textit{LKM} mediante la creación de un fichero virtual propio en este sistema, \texttt{/proc/fdev}.\\

El objetivo principal de esta fase fue comprobar que el módulo podía crear una entrada propia en \texttt{/proc}, asociarle funciones manejadoras (\textit{handlers}) de lectura y escritura, y responder de manera controlada a estas operaciones invocadas desde \textit{Espacio de Usuario}.\\

En primer lugar, fue necesario ampliar el conjunto de cabeceras incluidas en el módulo para poder trabajar con \texttt{/proc} y con las transferencias de datos entre \textit{Espacio de Usuario} y \textit{Espacio del Kernel}:

\begin{code}[h]
\begin{lstlisting}[language=C]
#include <linux/module.h>
#include <linux/init.h>
#include <linux/kernel.h>
#include <linux/proc_fs.h>
#include <asm/uaccess.h>
\end{lstlisting}
\caption[Cabeceras para el prototipo basado en \texttt{/proc}]{Cabeceras para el prototipo basado en \texttt{/proc}.}
\label{cod:proc_headers}
\end{code}

Se incorporan \verb|<linux/proc_fs.h>|, necesaria para crear y gestionar entradas dentro del sistema de ficheros virtual \texttt{/proc}, y por otro lado, \verb|<asm/uaccess.h>|, que importa funcionalidades como \verb|copy_to_user| y \verb|copy_from_user|, utilizadas para transferir información de forma controlada y segura entre la memoria del \textit{Kernel} y la memoria de \textit{Espacio de Usuario}.\\

A continuación, fue necesario definir una referencia a la entrada virtual y la tabla de operaciones asociada a dicha entrada en \texttt{/proc}:

\begin{code}[h]
\begin{lstlisting}[language=C]
static struct proc_dir_entry *ent;

static struct file_operations myops =
{
    .owner = THIS_MODULE,
    .read = myread,
    .write = mywrite,
};

static int simple_init(void)
{
    ent = proc_create("fdev", 0660, NULL, &myops);
    if (!ent)
        return -ENOMEM;
    printk(KERN_INFO "/proc/fdev created\n");
    return 0;
}

static void simple_cleanup(void)
{
    proc_remove(ent);
    printk(KERN_INFO "/proc/fdev removed\n");
}

module_init(simple_init);
module_exit(simple_cleanup);
\end{lstlisting}
\caption[Referencia a la entrada virtual y tabla de operaciones asociadas en \texttt{/proc}]{Referencia a la entrada virtual y tabla de operaciones asociadas en \texttt{/proc}.}
\label{cod:proc_entry_ops}
\end{code}

En este fragmento, el puntero \verb|ent| almacena la referencia a la entrada que se crea en \texttt{/proc} al cargar el módulo, permitiendo posteriormente su gestión y eliminación al descargarlo.
Por otro lado, la estructura \verb|myops| define la tabla de operaciones del fichero virtual a crear, asociando las acciones de lectura y escritura sobre dicho fichero con las funciones manejadoras (\textit{handlers}) \verb|myread| y \verb|mywrite|, respectivamente.
Además, el campo \verb|.owner = THIS_MODULE| vincula estas operaciones al propio módulo cargado.
Por último, \verb|simple_init| crea la entrada virtual \texttt{/proc/fdev} mediante \verb|proc_create| durante la carga del módulo, devolviendo \texttt{-ENOMEM} (\textit{no memory error}) si la entrada no puede crearse, mientras que \verb|simple_cleanup| la elimina con \verb|proc_remove| durante su descarga.\\

Un aspecto a resaltar en el desarrollo de los prototipos de \textit{Módulos del Kernel} es la conveniencia de usar \textit{static} para declarar variables y funciones que no se van a utilizar fuera del propio módulo, restringiendo así su visibilidad al código fuente del módulo y favoreciendo una mejor encapsulación del código, por lo que este criterio se mantendrá en el resto del proyecto.\\

Una vez creada la entrada al fichero virtual \texttt{/proc/fdev} y asociada su tabla de operaciones, se implementaron los \textit{handlers} de escritura y lectura de dicho fichero:

\begin{lstlisting}[language=C]
enum { BUFSIZE = 100 };

static int irq = 20;
static int mode = 1;

static ssize_t mywrite(struct file *file, const char __user *ubuf,
		       size_t count, loff_t *ppos)
{
	int num, c, i, m;
	char buf[BUFSIZE];

	printk(KERN_DEBUG "write handler\n");

	if (*ppos > 0 || count >= BUFSIZE)
		return -EFAULT;

	if (copy_from_user(buf, ubuf, count))
		return -EFAULT;

	num = sscanf(buf, "%d %d", &i, &m);
	if (num != 2)
		return -EFAULT;

	irq = i;
	mode = m;

	c = strlen(buf);
	*ppos = c;

	printk(KERN_DEBUG "written to /proc/fdev: %d bytes received from user\n", c);
	return c;
}

static ssize_t myread(struct file *file, char __user *ubuf,
		      size_t count, loff_t *ppos)
{
	char buf[BUFSIZE];
	int len = 0;

	printk(KERN_DEBUG "read handler\n");

	if (*ppos > 0 || count < BUFSIZE)
		return 0;

	len += sprintf(buf, "irq = %d\n", irq);
	len += sprintf(buf + len, "mode = %d\n", mode);

	if (copy_to_user(ubuf, buf, len))
		return -EFAULT;

	*ppos = len;

	printk(KERN_DEBUG "read from /proc/fdev: %d bytes sent to user\n", len);
	return len;
}
\end{lstlisting}
\captionsetup{type=code}
\captionof{code}[\textit{Handlers} con transferencia de datos entre \textit{Espacio de Usuario} y \textit{Espacio del Kernel}]{\textit{Handlers} con transferencia de datos entre \textit{Espacio de Usuario} y \textit{Espacio del Kernel}.}
\label{cod:proc_rw_handlers}

\

\

La función manejadora de escritura, \verb|mywrite|, recibe datos desde \textit{Espacio de Usuario} mediante \verb|copy_from_user|, trasladando \verb|count| bytes del \textit{buffer} de memoria de usuario (\textit{ubuf}) a un \textit{buffer} en memoria del \textit{Kernel} (\textit{buf}).
Análogamente, \verb|myread| devuelve datos hacia \textit{Espacio de Usuario} mediante \verb|copy_to_user|, trasladando \verb|len| bytes del \textit{buffer} en memoria del \textit{Kernel} (\textit{buf}) al \textit{buffer} de memoria de usuario (\textit{ubuf}).
Además, mediante el puntero de posición del fichero (\verb|*ppos|) se impuso una semántica de acceso de tipo \textit{single-shot}, evitando lecturas o escrituras repetidas sobre la misma apertura del fichero virtual.\\

En este ejemplo, los valores \verb|irq| y \verb|mode| se emplean como variables enteras de prueba para validar el intercambio de información entre \textit{Espacio de Usuario} y \textit{Espacio del Kernel}, permitiendo comprobar su correcta interpretación a partir de los datos recibidos mediante \verb|sscanf|, así como su formateo para la salida mediante \verb|sprintf|, y retornando errores como \texttt{-EFAULT} (\textit{bad address}) cuando no se cumplen dichas restricciones o cuando el formato de entrada no es el esperado.
Por otro lado, se define una constante de tipo enumerado \verb|BUFSIZE| para fijar de forma simple el tamaño máximo del \textit{buffer} temporal utilizado, en este caso 100 bytes, tanto para la lectura como para la escritura.\\
