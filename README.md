# SECURE REPORTING SYSTEM AGAINST USER-SPACE ROOTKITS

This repository contains the source code and documentation for a Bachelor's Degree Project focused on secure system monitoring in Linux environments affected by user-space rootkits.

The project designs and implements a Loadable Kernel Module (LKM) that generates a complete process list report of the system directly from kernel-space and authenticates it with HMAC-SHA256 by a secret symetric key. The goal is to provide the administrator with a trusted source of critical system information when all user-space and even monitoring tools, such as `ps`, `top` or `netstat`, may have been intercepted or manipulated by malware, specifically user-space rootkits which has persistence and concealment skills.

The final prototype transfers an output file descriptor from user-space to the kernel through a `/proc` interface. The LKM then writes an authenticated report containing running process metadata and a Base64-encoded HMAC tag. A user-level verifier recomputes the HMAC with the same symmetric key and checks whether the report has preserved its integrity and authenticity.

## Threat Model

The implemented mechanism is intended as a research prototype, not as a production-ready security product. It assumes a threat model where:

- User-space may be compromised by a rootkit.
- Kernel-space, the LKM and the boot trust chain remain intact.
- The administrator is trusted.
- The symmetric key is not available to the attacker.
- The kernel can write authenticated information into a file opened by the administrator.

Within this model, the prototype demonstrates how sensitive system information can be generated from a higher-privileged layer and verified afterwards from User Space.

## Main Components

### Kernel Module

Located in `src/LKM/crypto_fd.c`.

The LKM creates the `/proc/fddev` virtual entry and implements the read/write handlers used for communication with User Space. Its main responsibilities are:

- receiving a file descriptor from the administrator through `/proc/fddev`;
- resolving that descriptor inside Kernel Space;
- collecting running process metadata from kernel structures;
- protecting the task-list traversal with RCU;
- calculating an HMAC-SHA256 tag with the Linux Kernel Crypto API;
- encoding the resulting tag in Base64;
- writing the authenticated report to the handoff file.

The generated report includes fields such as kernel UID, local UID, UID namespace, kernel PID, local PID, PID namespace, GID and command path/name.

### Key Generation

Located in `src/LKM/gen_k_embedded.sh`.

This script generates a 64-byte symmetric key and converts it into a C header file:

- `key.bin`: raw binary key.
- `k_embedded.h`: C header containing `K` and `K_len`.

The embedded key is used both by the LKM and by the user-space verification utility. For real deployments, static embedding should be replaced by a stronger key-provisioning mechanism.

### File Descriptor Handoff Utility

Located in `src/mk_fhandoff.c`.

This user-space program creates or opens the handoff file and writes its file descriptor into `/proc/fddev`. This allows the kernel module to write the authenticated report into a file controlled by the administrator without trusting the content generated by User Space.

### HMAC Verification Utility

Located in `src/check_hmac.c`.

This program reads the generated handoff file, extracts the process report and the Base64 HMAC tag, recomputes the HMAC-SHA256 value locally with OpenSSL and compares both tags using constant-time comparison.

## Handoff File Format

The file generated by the kernel follows this structure:

```text
--KERNEL-PS--
UID_KERNEL  UID_LOCAL   UID_NS        PID_KERNEL  PID_LOCAL   PID_NS        GID        COMMAND
...

--HMAC--
<Base64-encoded HMAC-SHA256>
```

The HMAC is calculated over the process table generated in Kernel Space. The verifier extracts that same content, recomputes the tag and checks that both values match.

## Build Requirements

The prototype targets Linux systems with support for external kernel module compilation.

General requirements:

- Linux kernel headers matching the running kernel.
- GCC and Make.
- OpenSSL development libraries for the user-space verifier.
- Root privileges to load and unload the kernel module.

On Debian/Ubuntu-based systems, the user-space dependencies are typically covered by:

```bash
sudo apt install build-essential libssl-dev
```

Kernel headers depend on the running distribution and kernel version.

## Build and Run

> Loading kernel modules can affect system stability. Use a controlled testing environment.

1. Build the kernel module.

```bash
cd src/LKM
chmod +x gen_k_embedded.sh
./gen_k_embedded.sh
make
```

2. Load the module.

```bash
sudo insmod crypto_fd.ko
```

3. Check that the `/proc` entry exists.

```bash
cat /proc/fddev
```

4. Build the user-space utilities.

```bash
cd ..
gcc -Wall -Wshadow -Wvla -c mk_fhandoff.c
gcc -o mk_fhandoff mk_fhandoff.o

gcc -Wall -Wshadow -Wvla -g -c check_hmac.c
gcc -g -o check_hmac check_hmac.o -lcrypto
```

5. Generate the authenticated report.

```bash
./mk_fhandoff ./file_handoff /proc/fddev
```

6. Verify the report.

```bash
cat ./file_handoff
./check_hmac ./file_handoff
```

7. Unload the module.

```bash
cd LKM
sudo rmmod crypto_fd
```

Kernel logs can be inspected with:

```bash
dmesg -T | tail
```

(is not the only way)

## Security Notes

This prototype focuses on authenticity and integrity of the report generated by the kernel. It does not attempt to detect malware automatically, nor does it protect against kernel-space rootkits. If the kernel, the LKM or the key are compromised, the trust assumptions of the design no longer hold.

The project deliberately keeps the implementation small and explicit so that the communication path, HMAC generation and validation process can be studied and reproduced.
